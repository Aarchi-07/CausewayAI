How the filter strategy works

When a query arrives, the system does not search the full 30,000-call corpus blindly. It first asks Gemini Flash to parse the query into a structured search plan: what is the search_text for the embedding query, and what filters apply?

Filters operate on two levels. Domain is a hard constraint — a must-match condition in Qdrant's filtering API. If a query is clearly about banking, calls from telecom or retail are excluded before any embedding comparison occurs. This collapses the search space from 30,000 to ~5,000 candidates instantly.

Topic and outcome are soft constraints — should-match conditions that influence ranking without hard exclusion. A call matching the topic receives a scoring boost, but a call with a high cosine similarity to the query vector is not excluded just because its topic differs. This prevents the filter from being too aggressive in cases where the user's intent is exploratory.

The hybrid approach — hard domain filter + soft topic/outcome filter + cosine re-ranking — addresses the classic precision-recall tradeoff. Domain filtering gives precision for free (wrong-domain calls can never be right). Soft topic filtering maintains recall for edge cases. Embedding re-ranking restores semantic precision within the filtered set.

The embedding model

Every call in the corpus is embedded using google/embeddinggemma-300m, a 300M parameter embedding model that produces 768-dimensional vectors. These vectors encode the semantic content of the call's enriched summary and problem-resolution segment — not the raw transcript, which would dilute the signal with filler words and procedural language.

At query time, the same model embeds the reformulated search text. The cosine similarity between query vector and call vectors determines ranking within the filtered candidate set. Calls scoring below 0.4 cosine similarity are discarded regardless of filter match.

Storing 768-dimensional vectors for 30,000 calls requires building and maintaining the Qdrant collection. The collection holds two types of documents: inter_iit_conversations (one vector per call, used by the filter strategy) and inter_iit_knowledge_graph (one vector per graph node, used by GraphRAG for entry-point retrieval).

Strengths and failure modes

Why this strategy excels

ID Recall: 36.0% — the highest of the two strategies. Because the vector search is not constrained to graph adjacency, it surfaces a broader set of relevant calls. For general exploratory queries — "what issues appear in banking calls?" — this breadth is an advantage. The system returns diverse, relevant evidence without requiring the queried topic to be represented as a specific graph Concept node.

The strategy is also more robust to vocabulary variation. If a user asks about "connectivity problems" and the corpus uses "signal issues," the embedding similarity will still surface relevant calls — because both phrases occupy similar regions of the 768-dimensional semantic space. GraphRAG would fail here if "connectivity problems" doesn't map to a known Concept node.

Where it falls short

Faithfulness: 0.58. The trade-off for broader coverage is lower answer faithfulness. Because the retrieval is less structurally constrained, the LLM receives a heterogeneous evidence set — calls that are semantically similar to the query but not causally related to each other. This creates opportunity for the model to synthesise across unrelated evidence, producing answers that are coherent but partially fabricated.

The strategy also cannot perform multi-hop reasoning. A query connecting two different causal patterns requires the intersection of two evidence sets — a graph operation. The vector strategy can only retrieve by proximity to a single query vector, which means it cannot answer relational queries that GraphRAG handles natively.
