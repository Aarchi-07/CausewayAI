How do you know why a customer escalated? Current systems know it happened — they have the outcome flag, the supervisor transfer, the ticket. But the conversation that produced it, the exact chain of turns that pushed the interaction to breaking point, remains opaque. Multiply this by thirty thousand conversations and thirty million tokens, and you have a blindspot at industrial scale.

This is the problem CausewayAI was built to solve. Not to report outcomes — to explain them. The system accepts natural-language queries against a large corpus of customer-agent transcripts and returns structured, evidence-backed causal explanations, citing specific call IDs and dialogue spans. It sustains multi-turn dialogue, handles follow-up questions with full context, and does this without hallucinating by grounding every claim in retrieved evidence.

Background

Contact centre operations across industries — banking, insurance, retail, telecom, flight, hotel — share a structural problem: they produce enormous amounts of conversational data and extract very little analytical value from it beyond aggregate metrics. Escalation rate. Average handle time. CSAT score. These are outcomes.

The question operators actually need answered is different: what conversational patterns produce bad outcomes? What phrasing triggers an escalation? When does billing confusion become a churn risk? When does a repeat query signal a knowledge gap in the agent pool versus a systemic product failure?

Answering these questions requires reasoning across entire transcripts, not individual utterances. It requires connecting a churn signal in call 4,823 to a billing confusion pattern also present in calls 11,004, 17,390, and 22,771. It requires doing this in response to an analyst's natural-language question, in real time, with zero tolerance for fabrication.

This is where standard NLP pipelines and generic LLM usage both fail. A standard pipeline extracts features but cannot reason causally. A generic LLM generates plausible-sounding answers but cannot ground them in specific evidence from your corpus. You need something in between — a system with structured retrieval layered under a reasoning model.

The problem, precisely

The challenge framed the task in two parts. The first: given a natural-language query, generate a structured causal explanation with supporting evidence from the transcript corpus. The second: extend this to multi-turn dialogue — handle follow-up questions while maintaining the context of what has already been established.

Both tasks share the same hard constraint: every claim must be evidentially grounded. Explaining why escalations happen by citing specific call IDs is not the same as summarising a general trend. The former is auditable. The latter is guesswork dressed as analysis.

The corpus available was 30M+ tokens of customer-agent dialogue, spanning six domains, with speaker labels, turn indexing, and outcome flags. No pre-labelled causal spans. No structured annotation. Raw transcripts, at scale.

The core challenge: causal triggers in conversational data are not explicit. They are distributed across turns, often expressed through absence (no resolution after three queries), repetition (the same complaint in different words), or structural patterns (escalation always follows a specific billing-explanation failure mode). Extracting them requires both a retrieval system with high coverage and a reasoning layer with high faithfulness.

Our approach

We built a two-strategy retrieval engine. Both strategies share the same entry point — a natural-language query — and the same exit point — a streamed, grounded LLM answer. What differs is how they retrieve evidence. The choice of strategy is determined by the query type.

Step 1 — Label discovery and metadata enrichment

Raw transcripts carry no structured metadata. Before retrieval is possible, every call must be tagged with a domain, a topic, and an outcome. We did this in two stages to prevent a well-known failure mode: naming drift.

In the first stage, we sampled 40% of the corpus and sent batches to Gemini Pro with an open-ended prompt: identify the topic and outcome from this conversation. Across thousands of calls, Gemini Pro surfaced a consistent set of candidate labels — the Zipfian distribution of topics meant that 40% coverage captured the vast majority of the label space.

In the second stage, we fixed this schema and ran every remaining call through Gemini Flash with constrained classification — choose from this exact list of topics and outcomes. This prevented the model from coining synonyms ("Billing Dispute" vs "Bill Dispute" vs "Billing Error") and ensured every field in the metadata was deterministic and queryable.

Traditional NLP preprocessing — stopword removal, lemmatisation, stemming — was deliberately avoided. These transforms destroy conversational signal. Hesitations, repetitions, and informal agent phrasing are features, not noise. Stripping them degrades classification accuracy on this class of data.

Step 2a — GraphRAG: knowledge graph traversal

We constructed a knowledge graph from the enriched corpus. The graph has two node types: Call nodes containing the full transcript text, summary, and metadata; and Concept nodes serving as structural anchors for traversal. Edges connect calls to the topics and outcomes they instantiate.

When a query arrives, we embed it using embeddinggemma-300m (768 dimensions) and run a vector search against the graph's embedding index to find initial entry nodes. A domain filter is inferred from the query and used to restrict which Call nodes are eligible. Then graph traversal begins.

Three traversal modes handle different query structures:

  • Single-concept expansion: retrieve all Call nodes adjacent to a single Concept node.
  • Multi-concept intersection: for queries involving multiple dimensions ("billing confusion AND escalation"), find Call nodes that appear in the neighbourhood of all relevant Concepts simultaneously.
  • Shortest-path bridging: when no intersection exists, find the minimum-length path between Concept nodes and extract Calls along the path — connecting structurally related regions of the graph.

The retrieved calls are assembled into a context block and passed to Gemini Flash, with an explicit instruction to cite Call IDs for every claim. The response is streamed back via Server-Sent Events.

Step 2b — Vector RAG: hybrid prefilter search

The second strategy skips the graph and queries Qdrant directly. The query is parsed by Gemini Flash into a structured search plan: a search_text for the embedding query, and filter conditions. Domain filters are applied as hard must-match conditions in Qdrant's filter API. Topic and outcome filters are applied as soft should-match conditions — influencing ranking without hard exclusion.

This produces a hybrid retrieval: the prefilter trims the search space dramatically (from 30,000 calls to hundreds or fewer), then the embedding similarity selects the most semantically relevant subset. Top-40 results are retrieved; those below a 0.4 cosine score are discarded. The surviving evidence is assembled and passed to the same LLM generation step.

Step 3 — Multi-turn context

Follow-up queries ("What about the telecom cases?" following a question about billing escalations) are handled by a query reformulation step. The chat history is passed to Gemini Flash with a single instruction: rewrite the latest query as a standalone search query, resolving pronouns and implicit references. This allows the retrieval step to operate statelessly while preserving conversational continuity at the application layer.

The backend is a single FastAPI application with a /chat endpoint that returns an SSE stream. Each event in the stream carries a typed payload: status (pipeline stage), concepts (inferred filters or graph entry nodes), sources (retrieved call records), and token (streamed LLM output). The frontend consumes these incrementally, rendering sources as cards before the full answer arrives.

Evaluation

We evaluated both strategies using the RAGAS framework across three metrics. ID Recall measures whether the correct call records appear in the retrieved evidence set. Faithfulness measures whether the generated answer is supported by the retrieved evidence — a faithfulness of 1.0 means every claim is traceable to a source; 0.0 means the model fabricated. Answer Relevancy measures whether the answer addresses the question asked.

The results reveal a fundamental tradeoff. GraphRAG achieves substantially higher faithfulness — 0.74 versus 0.58 — meaning its answers are more tightly grounded in the retrieved evidence and carry a significantly lower hallucination risk. This makes it the correct choice for compliance-sensitive queries, root cause analysis, and any use case where an incorrect citation has consequences.

Vector RAG achieves higher recall and relevancy. Because it searches by semantic similarity rather than strict graph adjacency, it surfaces a broader set of relevant calls (36% vs 30.8% recall) and produces answers that more directly address the surface-level question. For exploratory queries — "what kinds of issues appear in telecom churn calls?" — this broader coverage is an advantage.

Key finding: neither strategy is universally superior. A production deployment would route compliance and root-cause queries to GraphRAG and general information retrieval to Vector RAG. The routing decision can itself be made by a lightweight query classifier.

What we achieved

This is what the system is capable of doing, against a corpus no existing tool was processing analytically before:

  • Answer the question "Why are escalations happening in banking calls?" with citations to specific call IDs and quoted transcript spans.
  • Perform multi-hop reasoning: "What connects fraud reports in insurance to churn signals in telecom?" — finding shared structural patterns via shortest-path graph traversal.
  • Sustain multi-turn dialogue: follow-up questions maintain context without reprocessing the entire history.
  • Operate at 30M+ token scale with sub-10-second response latency via streaming.
  • Achieve faithfulness scores competitive with purpose-built RAG systems, without any fine-tuning of the underlying LLM.

The system is fully containerised. A single docker compose up starts the FastAPI backend with the knowledge graph pre-loaded and Qdrant initialised. The frontend is a static HTML/JS client that connects to the /chat SSE endpoint and renders evidence sources in real time.

Limitations and future work

GraphRAG is brittle when initial entity resolution fails. If the embedding search does not find a relevant Concept node as an entry point — because the query uses phrasing not represented in the graph's vocabulary — traversal never begins, and the answer falls back to the evidence found by direct vector search on Call nodes alone. Improving robustness here, through synonym expansion or a more capable entity linker, is the most important near-term improvement.

The current graph was built from a static corpus snapshot. In a production system the graph would need to be incrementally updated as new calls arrive — a non-trivial engineering challenge that requires either periodic re-indexing or an online graph update protocol.

Finally, the evaluation framework, while rigorous, was constrained by the absence of a gold-standard causal annotation set. ID Recall is measured against a synthetically constructed ground truth. A purpose-built evaluation dataset, with human-annotated causal span labels, would allow much more precise measurement of the system's core capability.
