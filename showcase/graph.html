Why a graph, not just a database

The fundamental limitation of vector search — even good vector search — is that it retrieves by similarity, not by causal structure. A query about billing escalations retrieves calls that are semantically similar to "billing escalation." But it cannot answer: which calls connect billing confusion to churn risk? That requires traversal across a structured relation, not a similarity ranking.

A knowledge graph encodes structure explicitly. When a billing complaint call also resulted in a churn risk signal, that relationship exists as an edge. When five different domain concepts all share a cluster of call nodes, that intersection is queryable in constant time. Graph traversal can find causal chains that embedding similarity simply cannot surface.

The 88MB graph artifact encodes every call as a node with its full text, summary, and domain. Every topic and outcome is a Concept node. An edge between a Call node and a Concept node means that call instantiates that concept. This structure allows multi-hop causal reasoning that would otherwise require scanning the entire corpus.

Three traversal modes

I

Single-concept expansion

The simplest case: a query maps to a single Concept node. The system retrieves all Call nodes adjacent to that concept in the undirected graph. If you ask about fraud reports in banking, the system finds the "Fraud Report" Concept node and returns every call connected to it within the banking domain. Fast, high precision for well-defined queries.

II

Multi-concept intersection

A query involves two or more distinct dimensions — "billing confusion AND escalation." The system builds a neighbourhood set for each Concept node and computes the intersection: calls that appear in both concept neighbourhoods simultaneously. These are the calls where both conditions co-occurred. If no intersection exists, the system falls through to shortest-path bridging.

III

Shortest-path bridging

When no direct intersection exists between two concept neighbourhoods, the system finds the minimum-length path between them in the undirected graph. Paths of length ≤ 4 are expanded: Call nodes along the path are collected, and Concept nodes along the path spawn their adjacent Call nodes. This allows the system to surface structurally related regions of the corpus even when no single call instantiates every queried concept.

Strengths and failure modes

Strengths

Faithfulness: 0.74. The highest score of either strategy. Because the graph traversal only returns calls that are structurally connected to the queried concepts, the evidence set is always coherent — the LLM cannot stray far from what the graph selected. For compliance-critical queries, this is the correct strategy.

The graph also handles multi-hop reasoning natively. A request like "what connects fraud patterns in insurance to churn signals in telecom?" is answerable by finding shortest paths between two Concept nodes in different domain subgraphs — a query type that vector search cannot resolve.

Failure modes

GraphRAG is brittle at the entry point. If the initial vector search fails to retrieve a relevant Concept node — because the query uses vocabulary not present in the graph — traversal never begins. The system falls back to the Call nodes found by direct embedding search, losing the structural advantage entirely.

The graph is also static. It was built from a corpus snapshot. Calls added after graph construction are indexed in Qdrant but absent from the graph, creating a divergence between the two retrieval paths over time. Incremental graph updates are the primary open engineering problem.
